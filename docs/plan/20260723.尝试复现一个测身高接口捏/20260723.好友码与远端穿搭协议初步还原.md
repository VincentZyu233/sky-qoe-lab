# 好友码与远端穿搭协议初步还原

## 当前结论

`sky.cesgao.cn` 的公开源码只实现了第三方 PoW 和转发客户端，没有包含网易或 TGC 的登录、好友码或穿搭协议喵。

当前 Sky 国服客户端中已经静态确认以下两段官方能力喵：

1. 好友邀请由 `create_invite`、`check_invite`、`accept_invite` 和 `delete_invite` 四个接口组成喵。
2. 远端玩家穿搭由独立的 `get_remote_outfit` 接口读取，请求字段为 `target` 喵。

因此最可能的测量链路是先用好友码 token 调用 `check_invite`，从响应中取得目标玩家标识与身高相关资料，再用目标标识调用 `get_remote_outfit` 喵。

目前尚未静态证明 `check_invite` 响应中目标标识的确切字段名，也尚未证明第三方接口的 `action=delete` 是调用 `delete_invite`、`accept_invite`，还是第三方服务自己的任务清理选项喵。

## 第三方接口

第三方客户端确认的调用如下喵：

```text
GET  https://sky.cesgao.cn/web/v1/pow/challenge
POST https://sky.cesgao.cn/web/v1/query
```

查询请求形状如下，实际好友码和 PoW 值不得写入仓库喵：

```json
{
  "query": "XXXX-XXXX-XXXX",
  "action": "delete",
  "pow_token": "...",
  "timeout": 60
}
```

第三方响应格式显示它最终能返回 `uuid`、`body_size`、`height_value`、`current_height`、`max_height`、`min_height` 与 10 个穿搭槽位喵。

这些字段是第三方服务的输出格式，不应直接假设为 Sky 官方 JSON 字段名喵。

## HAR 结论

已脱敏分析的 HAR 共包含 216 条记录与 3 个 Sky 游戏域名端点喵：

| 方法 | 主机与路径 | 结论 |
| --- | --- | --- |
| `GET` | `live-queue-sky-merge.game.163.com/queue` | 登录队列喵 |
| `POST` | `live-as-sky-merge.game.163.com/account/gameconfig/get_config` | 游戏配置喵 |
| `POST` | `live-gms-sky-merge.game.163.com/gms_cmd` | 进入游戏流程喵 |

`get_config` 响应中出现了 `nt_outfit_defs_stripped` 与 `force_rebuild_cape_spine`，可确认该主机属于真实游戏服务链路喵。

这份 HAR 没有出现 `friend`、`invite`、`get_remote_outfit` 或身高查询请求，因此不能直接用于恢复好友码协议喵。

网易精灵的 `sprite.gameyw.netease.com` 请求只涉及内容、知识搜索和用户中心，与游戏好友码无关喵。

## 已确认官方请求

以下 RVA 均属于模块基址 `0x140000000`、PE 时间戳 `0x6A582C8E` 的当前构建喵。

| 请求 | 路由字符串 RVA | 路由 getter RVA | 序列化函数 RVA | 请求字段 | 响应解析 RVA |
| --- | ---: | ---: | ---: | --- | ---: |
| Create invite | `0x1F30956` | `0x1922F50` | `0x1922F60` | `invite_nickname` | `0x1923020` |
| Delete invite | `0x1F3093F` | `0x1923540` | `0x1923550` | `token_id` | `0x1923610` |
| Check invite | `0x1F30929` | `0x1923C90` | `0x1923550` | `token_id` | `0x1923CA0` |
| Accept invite | `0x1F308C1` | `0x19241A0` | `0x19241B0` | `token_id`, `invite_nickname` | `0x19243A0` |
| Remote outfit | `0x1EE7A49` | `0x1600530` | `0x1600540` | `target` | `0x1600580` |

对应路由如下喵：

```text
/account/create_invite
/account/delete_invite
/account/check_invite
/account/accept_invite
/account/get_remote_outfit
```

`check_invite` 的 vtable 序列化槽位直接指向 `delete_invite` 的 `0x1923550`，所以两者都以 `token_id` 为请求字段，这是静态确认而不是推测喵。

`accept_invite` 的请求对象在 `+0x68` 保存 token，在 `+0x78` 保存昵称喵。

`get_remote_outfit` 的请求对象在 `+0x68` 保存目标标识，并序列化为 `target` 喵。

客户端还包含 `AccountRemoteOutfitRequest request failed. target:` 和 `Response target desynced` 日志，说明响应会校验返回目标是否与请求目标一致喵。

## 外部好友接口

客户端另有以下接口喵：

```text
/account/get_external_friends
/account/accept_external_friend
/account/confirm_external_friend
/account/invite_external_friend
```

它们对应 `AccountAcceptExternalFriendRequest`、`AccountConfirmExternalFriendRequest` 和 `AccountInviteExternalFriendRequest`，更可能用于 Steam 或其他平台好友同步，不应与临时好友码四接口混淆喵。

## 分析脚本

脚本按执行顺序保存在 `test/20260723` 喵：

1. `01_har_endpoint_inventory.py` 生成不含请求头、Cookie、查询值、正文值或动态路径标识的 HAR 端点清单喵。
2. `02_extract_sky_protocol_strings.py` 从模块布局 dump 中提取关键词字符串并输出偏移、RVA 和 VA 喵。
3. `03_find_rip_xrefs.py` 使用 Capstone 查找字符串或函数 RVA 的 RIP 相对引用与直接立即数引用喵。
4. `04_analyze_pe_functions.py` 使用 `.pdata` 获取函数边界，并解析函数内引用的静态字符串与直接调用喵。

`03` 与 `04` 需要 `capstone==5.0.9`，当前只安装在已忽略的 `test/20260723/.venv` 中喵。

## 已遇到问题

不要让 CE 对完整 Sky 模块调用 `DissectCode:dissect()` 或 `createRipRelativeScanner()` 喵。

在当前约 50 MB 模块上，前者超过 180 秒、后者超过 30 秒仍未返回，并会占住 CE Bridge 的单请求线程喵。

两次实验均未设置断点，Sky 进程始终显示 `Responding=True`，最终通过只重启 CE、保持 Sky PID 不变并重新附加恢复了桥喵。

全模块 xref 现在统一使用离线 `Sky_dump.bin` 与 `03_find_rip_xrefs.py`，CE 仅用于小范围内存读取和运行时验证喵。

## 下一步

1. 反查 `check_invite` 请求对象的构造调用点，确认 12 位好友码如何转换或写入 `token_id` 喵。
2. 恢复 `check_invite` 响应对象的成员布局，定位目标玩家 ID、体型值与身高参数喵。
3. 恢复 `get_remote_outfit` 响应对象布局，确认 10 个槽位 ID 与现有本地穿搭枚举是否完全一致喵。
4. 在有专用测试好友码时做一次定向抓包或窄范围函数插桩，分别观察 `check`、`remote outfit` 与 `delete/accept` 的顺序喵。
5. 协议确认后再实现本地 HTTP/命名管道服务，避免当前阶段把第三方字段误当作官方字段喵。
